# P4 BLE: advertising + GATT telemetry verification (2026-06-01)

Verified the M5StickS3 P4 BLE build (gated `CONFIG_APP_BLE`) over the air from the
development PC using `bleak` (Windows BLE), after the phone GUI proved unreliable
and intermittent.

## HW-011 - advertising + manufacturer telemetry

Scanned "StickS3" (`70:04:1D:DB:AB:BA`): 26-74 adverts per 8-15 s window
(~3-5 Hz), RSSI -44..-46. Manufacturer data (company id `0xFFFF`, SIG
test/internal) decodes to a packed little-endian payload:

    uptime_s (u32) | bat_mv (s16) | accel_mg x/y/z (s16)

Observed live: uptime increments; battery 4176-4188 mV (matches HW-010 /
issue #4 ~4180 mV); accel Z ~ +1020 mg = +1.02 g (gravity, board flat), X/Y near
zero. Values track reality.

## HW-012 - connectable GATT read + notify

Connected from the PC; READ of the telemetry characteristic
(`a3b1f001-5e3a-4c7d-9f2a-7b1c5e3a4c7d`) returned valid data; subscribed and
received 6 notifications at ~1 Hz with live (incrementing) values.

## Bug found on hardware, then fixed

Initially the device stopped advertising after the first connect/disconnect cycle
(the "intermittent" BLE the phone showed) - a runtime bug the build and two code
reviews did not catch. Root cause: connectable advertising is stopped by the
controller on connection, and the disconnect callback did not restart it; calling
`bt_le_adv_start()` inline in the callback returns -EAGAIN before the connection
context is released. Fixed by deferring the restart to a `k_work` (system
workqueue) - `app/src/ble.c`. After the fix, a re-scan saw 28 adverts in 8 s
immediately after a disconnect: advertising resumes correctly.

## Default build stays blob-free

The default board build sets no `CONFIG_BT` (FLASH ~142 KB, CI-green); BLE is only
built via `app/overlay-ble.conf` and requires `west blobs fetch hal_espressif`.

## Tooling

PC-side scripts (local, off-repo): `ble_scan.py` (scan + decode), `ble_connect.py`
(connect + read + notify). Phone nRF Connect also works, but the PC path is
scriptable and decodes the payload directly.
